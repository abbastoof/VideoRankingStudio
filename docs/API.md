# REST API

The HTTP surface exposed by `apps/api` (Fastify). This document is the
reference for callers — humans reading the code, teams integrating, and the
`@vrs/sdk` maintainers. It intentionally stays high-level; the authoritative
per-route schema is generated by Fastify and served as OpenAPI.

## Base URL

- Development: `http://localhost:4000`
- Production: the value of `API_URL` in the deployed environment

Every non-trivial route is versioned under `/v1/…`. Unversioned paths
(`/health`, `/`) are infra endpoints only and not part of the product API.

## Interactive docs

In non-production builds the API mounts `@fastify/swagger-ui` at `/docs`.
Every route registered with a Fastify schema (all of them, as of this
writing) is discoverable there. The OpenAPI JSON is at `/docs/json`.

## Authentication

The API uses session cookies. There are no bearer tokens for first-party
callers — the browser and the SDK both work off cookies.

- `POST /v1/auth/otp/request` — request a one-time email code.
- `POST /v1/auth/otp/verify` — verify the code, receive `vrs_session` +
  `vrs_session_refresh` cookies.
- `POST /v1/auth/refresh` — rotate the access cookie against the refresh
  cookie.
- `POST /v1/auth/signout` — end the current session.
- `GET  /v1/auth/session` — inspect the current session.
- `GET  /v1/auth/google/authorize` and `.../callback` — OAuth entry points.

Cookies are `HttpOnly`, `SameSite=lax` by default, and `Secure` when
`SESSION_COOKIE_SECURE=true`. The web app never receives raw tokens.

### Session freshness cache

`requireAuth` short-caches session freshness in Redis with a 60-second TTL
plus a per-user tombstone on revoke-all. See `session-cache.service.ts` for
the exact invariants.

## Error envelope

Every non-2xx response uses the same shape:

```json
{
  "error": {
    "code": "PROJECT_NOT_FOUND",
    "message": "Project not found",
    "requestId": "0f5e3a4b-…",
    "details": { "…": "…" }
  }
}
```

- `code` is a stable machine-readable identifier from `@vrs/types.ErrorCode`.
  New codes are additive; existing codes are never renamed.
- `message` is safe to display to end users. 5xx errors always redact to a
  generic message; the details go to logs and Sentry.
- `requestId` matches the `X-Request-Id` response header — the same ID
  Sentry breadcrumbs use.
- `details` is optional and route-specific. Validation errors expose a
  `details.issues[]` array with `{ path, message, code }` per issue.

See `apps/api/src/lib/errors.ts` for the full set of typed constructors.

## Pagination

List endpoints follow a cursor convention:

```json
{
  "items": [ { "id": "…", "…": "…" } ],
  "nextCursor": "opaque-string-or-null"
}
```

- Pass `?cursor=<value>&limit=<n>` on subsequent calls.
- `nextCursor` is `null` when there is nothing more to fetch.
- Cursors are opaque — do not decode them; they are not guaranteed stable
  across releases.

## Idempotency

State-changing methods (`POST`, `PATCH`, `PUT`, `DELETE`) accept an
`Idempotency-Key` header. Behavior:

- First call with a given `(scope, method, path, key)` runs the handler and
  caches the response for 24 hours.
- Subsequent calls with the same key **and** matching request body replay
  the cached response verbatim.
- Same key with a different body → `422 UNPROCESSABLE_ENTITY`. This is a
  client bug and we surface it aggressively.
- Concurrent calls with the same key while the first is in flight →
  `409 CONFLICT`.

Scope is the authenticated user id when a session is present, otherwise the
IP address. Details in `apps/api/src/middleware/idempotency.ts`.

## Rate limits

Global default: `RATE_LIMIT_API_PER_MINUTE` (defaults to 120/min per user or
IP). A few routes override:

- `POST /v1/auth/otp/*` — `RATE_LIMIT_AUTH_PER_MINUTE` (default 10/min)
- `POST /v1/public/contact` — 5 requests per 10 minutes per IP
- Billing webhook — 1000/min (Stripe delivers bursts on backfills)

Exceeding a limit returns `429 TOO_MANY_REQUESTS` with a `Retry-After`
header.

## Public (unauthenticated) endpoints

These are safe to hit without a session and are used by the marketing site:

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/v1/public/contact` | Contact form submissions |
| `GET`  | `/v1/public/status`  | Component-level health snapshot |
| `GET`  | `/v1/public/version` | Build SHA and environment |
| `GET`  | `/v1/billing/plans`  | Public plan catalog |
| `GET`  | `/health`            | Liveness probe |
| `GET`  | `/health/ready`      | Readiness with dependency checks |

Public endpoints do not accept `Idempotency-Key` and rate-limit aggressively
by IP.

## Route index

The API is split into route modules under `apps/api/src/routes/`. Each
module registers a Fastify plugin.

| Prefix | Purpose | Auth |
| --- | --- | --- |
| `/v1/public`     | Contact, status, version | none |
| `/v1/auth`       | OTP, refresh, sign-out, Google OAuth | mixed |
| `/v1/users`      | Profile, sessions, self-deletion | required |
| `/v1/projects`   | Projects, tracks, clips, timeline | required |
| `/v1/projects/:id/generate/*` | AI generation entry points | required |
| `/v1/projects/:id/transcripts` and `.../captions` | Transcripts + captions | required |
| `/v1/uploads`    | Multipart upload orchestration | required |
| `/v1/assets`     | Asset CRUD | required |
| `/v1/templates`  | Template catalog | required |
| `/v1/voices`     | Voice catalog + cloning | required |
| `/v1/billing`    | Plans, subscription, invoices, portal, cancel, webhook | mixed |
| `/v1/publish`    | Publish targets + jobs | required |
| `/v1/ranking`    | Ranking workflow | required |
| `/v1/support`    | Support tickets | required |
| `/v1/notifications` | In-app notifications | required |
| `/v1/insights`   | Analytics endpoints | required |
| `/v1/admin`      | Admin console | required (ADMIN role) |
| `/v1/internal`   | Worker → API callbacks | `INTERNAL_SERVICE_TOKEN` header |
| `/v1/ws`         | WebSocket entry points | required |

## Versioning

The API is currently at `v1` and expected to remain there through initial
launch. Breaking changes will land under `/v2/…` alongside `/v1/…` with a
deprecation timeline announced in the changelog. Additive changes (new
fields, new endpoints, new error codes) are not versioned.
